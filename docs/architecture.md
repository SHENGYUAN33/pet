## 2. 可愛元素設計重點

- **Persona Agent 輸出規格**：`pet_name`、`personality_tags`、`voice_style`、`catchphrase`、`cute_moments`（供下游剪輯層標記高光片段）
- **鏡頭類型標記**：Storyboard Agent 需標註每個鏡頭 `shot_type`（例如 `cute_highlight`），讓剪輯層套用對應模板（慢動作、貼圖、卡點）
- **可愛感落地技術點**：慢動作/凍格特寫、可愛貼圖疊加（肉球/愛心/星星）、暖色調 LUT、卡通字幕動畫、BGM 節奏對齊剪輯點(beat-sync cut)

## 3. 旁白（寵物自我介紹）Pipeline

```
寵物基本資料 + 個性標籤 (Identity Card)
        ↓
Persona Agent → 人格化第一人稱文案
        ↓
Script Agent → 依版型切分時間軸：
   0-3s   開場鉤子（例如「嗨～我是豆花！」）
   3-20s  個性/故事介紹（對應互動片段）
   20-27s 領養呼籲 CTA
   27-30s 收容所資訊/QR code
        ↓
TTS（依 voice_style 選聲線：童聲/元氣/慵懶等）
        ↓
音訊混合層：旁白音量 ducking（BGM 自動降低）
```

TTS 選型：Coqui XTTS-v2 / F5-TTS（開源，zero-shot voice cloning + 情緒控制）；
ElevenLabs（商用，情緒表現與中文自然度目前最佳）。建議同一支影片可雙版本產出供人工審核層 A/B 挑選。

## 4. Provider 路由邏輯（設計草案）

```
Router 判斷依據（依序評估）：
1. 內容敏感度：涉及真實寵物臉部/個資 → 優先本地開源
2. 時效性：急件 → 商用 API；批次任務 → 開源排隊處理
3. 成本上限：單支影片成本 cap，超過自動降級開源模型
4. Fallback：商用 API 失敗/逾時 → 自動切開源模型重試
```

## 5. 開發階段對照

| 階段 | 目標 | 範圍 |
|---|---|---|
| P0 原型 | 驗證"上傳→出影片"可行 | 手動選片 + 開源TTS + FFmpeg模板 |
| P1 MVP | 單一收容所可用 | 資料處理層 + 簡化內容策劃 + 開源優先生成 + 基本審查 |
| P2 多代理協作 | 完整協作 | 六 Agent 全上線 + Provider Router |
| P3 品質與安全 | 降低人工負擔 | VLM一致性/事實查核/合規檢查 + 人工審核UI |
| P4 發布與追蹤 | 串接社群+數據回饋 | 平台API、A/B測試、儀表板 |
| P5 持續優化 | 數據反饋優化 | 成效與影片特徵關聯分析 |
