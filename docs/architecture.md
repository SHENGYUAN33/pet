# 系統架構規格書（Master Spec）

本文件是專案的權威架構規格，整合了原始架構圖規劃與後續設計討論後的最終版本。`CLAUDE.md` 僅放摘要與開發規範，細節一律以本文件為準。

> 核心定位：這不是「文字轉影片工具」，而是一套以**寵物真實資料治理**為核心，串接
> 資料整理 → 個性分析 → 腳本 → 分鏡 → 影音生成 → 審核 → 發布 → 成效回饋
> 的短影音生產平台。競爭力來自「真實資料治理＋人格化敘事＋可控分鏡＋人工審核＋領養成效閉環」，不是單一影片模型。

## 2. 生成策略：鏡頭級生成，而非一鏡到底

**不直接生成一整支 30 秒影片**，而是拆成 **5～7 個短鏡頭（每個 3～6 秒）**，各自獨立生成後再由影容製作層剪輯合成。

理由：
- 寵物外觀一致性更容易控制（每個鏡頭獨立做一致性檢查）
- 失敗只需重生「單一鏡頭」，不必整支重跑，大幅降低生成成本
- 旁白節奏可精準對齊分鏡時間軸

### 標準分鏡模板（30 秒版）

| 時間 | 敘事目的 | 畫面 | 旁白範例 |
|---|---|---|---|
| 0–3s | Hook | 看鏡頭、歪頭、伸手 | 「嗨！先別滑走，我等你好久了！」 |
| 3–8s | 基本介紹 | 正面照、走動、玩耍 | 「我是豆豆，兩歲，超愛撒嬌的小男生。」 |
| 8–14s | 個性亮點 | 玩球、翻肚、親近人 | 「我最喜歡陪人玩球，也會安靜坐旁邊。」 |
| 14–20s | 生活想像 | 居家/散步/窗邊情境 | 「下班回家時，我會第一個跑來迎接你。」 |
| 20–25s | 領養條件 | 真實照片＋資訊卡 | 「我已完成健康檢查，也會乖乖用尿墊。」 |
| 25–30s | 情感 CTA | 注視鏡頭、名字與聯絡資訊 | 「可以給我一個家嗎？點擊了解我的故事。」 |

15 秒版：保留 Hook + 個性亮點 + CTA 三段即可，由 Script Agent 依 `duration` 參數自動壓縮，而非另外重寫。

### 三種腳本敘事風格（一次產生 3 版供人工挑選，不直接發布單一版本）

| 風格 | 適用對象 | 範例語氣 |
|---|---|---|
| 萌系自我介紹 | 活潑、幼齡、外觀討喜 | 「我是奶茶，專長是歪頭、撒嬌，還有偷偷把你的心帶走。」 |
| 溫暖故事型 | 成犬/成貓、有特殊經歷 | 「我曾經不知道家是什麼，但我還是相信有一天會有人叫我的名字。」 |
| 反差幽默型 | 個性鮮明、表情豐富 | 「大家都說我看起來很兇，其實我連吸塵器都怕。」 |

## 3. 寵物 Profile 與 Identity Card

每隻寵物建立一份結構化 Profile，作為全系統唯一事實來源（single source of truth）；Fact-check Agent 與 Compliance Agent 都以此為比對基準，**任何腳本內容不得與 Profile 矛盾或捏造**。

完整 JSON 範例見 [docs/schemas/pet_profile.example.json](schemas/pet_profile.example.json)，關鍵欄位分類：

- **基本資料**：`pet_id`, `name`, `species`, `breed`, `sex`, `age`, `size`, `location`
- **健康狀態**：`vaccinated`, `neutered`, `microchipped`
- **標籤分層**（不可混為一談）：
  - 吸引力標籤：可愛、撒嬌、聰明、穩定
  - 生活適配標籤：公寓、家庭、長者、兒童、多寵家庭
  - 照護需求標籤：運動量、醫療需求、分離焦慮
  - **限制條件**（不可因廣告目的而隱藏，只能用溫和方式呈現）：不親貓、怕生、不適合幼童等
- **多媒體與授權**：素材清單、拍攝日期、`usage_license_status`、`ai_extension_allowed`
- **外觀特徵（Pet Identity Card 核心）**：毛色、花紋、耳型、眼睛顏色、體型、項圈/配件、特殊辨識點 —— 提供給所有影像生成模型作為一致性錨點

### 個性與賣點萃取原則

AI 不應把所有標籤塞進影片，而要萃取**最真實且最有吸引力的 2～3 個賣點**，並強制保留必要揭露：

```
原始標籤：親人、活潑、愛玩球、會坐下、已結紮、不親貓、喜歡散步

→ 主要賣點：① 非常親人主動靠近 ② 喜歡互動適合戶外飼主 ③ 已完成基本醫療處置
→ 必要揭露：不適合與貓共居（必須出現在影片中，不可省略）
```

## 4. 腳本生成引擎

輸入：Pet Profile ＋ 素材分析結果 ＋ 目標平台 ＋ 目標受眾 ＋ 風格 ＋ 語言 ＋ 長度限制

輸出：**結構化 JSON**（非純文字文案），逐鏡頭包含 `visual_source` / `visual_instruction` / `narration` / `subtitle` / `music_mood`。完整範例見 [docs/schemas/script.example.json](schemas/script.example.json)。

### 生成規則（Script Agent 的硬約束，須寫入 prompt 與後驗規則引擎）

- 30 秒中文旁白約 90～130 字；每句不超過 18～22 字
- 前 3 秒必須出現寵物、動作與吸睛句
- **不虛構**健康狀況、技能或救援故事（Fact-check Agent 逐句比對 Profile）
- 避免悲情操弄／死亡倒數／威脅式文案
- CTA 必須具體（例如「點擊了解豆豆的故事」而非空泛呼籲）
- 必要照護需求/限制條件不得省略
- 旁白語氣需對應 Persona（不是所有寵物都用幼兒語氣）

## 5. 影音生成三策略（可混合使用，建議比例見第 9 節）

### 策略 A：真實影片優先（預設、最推薦）
真實影片剪輯 ＋ 照片動態化補位 ＋ 字幕 ＋ 旁白 ＋ 音樂 ＋ 貼圖。外觀最真實、失敗率低、成本可控、保留機構信任感。

### 策略 B：照片轉動態影片（Image-to-Video）
僅當寵物缺少可用影片素材時使用。
- **適合**：眨眼、歪頭、輕微搖尾、呼吸、看鏡頭、緩慢前進
- **不建議**：複雜奔跑、與人激烈互動、大幅轉身、多寵物交互、與原照差異過大的場景
- 技術：商用可用 Runway（非同步 Image-to-Video API）、Google Veo（支援參考主體圖片維持外觀一致）；開源見第 7 節

### 策略 C：可愛幻想場景
例如寵物穿小領巾坐咖啡店、幻想露營、疊加愛心/星星特效。**必須在影片標示「部分畫面由 AI 創意生成」**，且幻想畫面不得用來暗示寵物的實際體型、行為能力或居住狀況。

## 6. 旁白與聲音系統

### 聲音角色設計（Persona Agent 輸出的一環）
不只分男聲/女聲，建議設計成人格化聲線：活潑小男孩感、甜美小女孩感、溫柔療癒、呆萌慢熟、沉穩大叔、高冷貓系、台灣自然口語（國語/台語/中英雙語）。**不得模仿未授權真人聲音**。

### TTS Pipeline
```
腳本文字 → 斷句與情緒標記 → TTS 生成 → 時長測量
→ 若超出鏡頭時長：腳本自動縮句或調整語速（不是硬切）
→ 音量正規化 → 與畫面對時
```

- 開源：Coqui XTTS-v2、F5-TTS（zero-shot voice cloning + 情緒控制）、GPT-SoVITS、Piper（輕量）
- 商用：ElevenLabs（情緒表現與中文自然度目前最佳）、Azure/Google TTS
- 建議同一支影片可雙版本產出（商用/開源）供人工審核 A/B 挑選，同時累積資料供未來微調自有語音模型

### 音訊分層與混音原則
最終影片至少含：旁白主聲道、背景音樂、環境音、可愛提示音（鈴鐺/肉球/啵聲）。
- 旁白永遠最清楚，音樂於旁白期間自動降低（ducking）
- 避免尖銳音效
- 輸出前做 Loudness Normalization
- 音樂需有合法商用授權（開源：MusicGen 生成 15-30s loop）

## 7. AI Provider 策略：商用 API × 開源模型並列

所有 AI 呼叫一律經過 **Provider Adapter** 統一介面，業務程式碼禁止寫死特定廠商 SDK。介面設計參考 [docs/reference/provider_adapter.py](reference/provider_adapter.py)（`VideoGenerationProvider`、`TTSProvider`、`LLMProvider`、`VLMProvider` 等抽象基底類別）。

| 功能 | 商用 API | 開源方案 |
|---|---|---|
| LLM（Profile 摘要/腳本/分鏡/QA 判斷） | Claude / GPT-4o | Llama 3.x、Qwen2.5、Mistral（vLLM 部署） |
| VLM（照片內容/表情/姿勢/場景/品質分析） | GPT-4V / Claude Vision | Qwen2-VL、LLaVA-NeXT、CogVLM |
| 影片生成 I2V/T2V | Runway、Google Veo、Kling、Luma | Stable Video Diffusion、CogVideoX、Open-Sora、Mochi 1 |
| 圖像生成/增強 | Midjourney API、GPT-Image | SDXL / Flux.1、Real-ESRGAN |
| TTS 旁白 | ElevenLabs、Azure/Google TTS | Coqui XTTS-v2、F5-TTS、GPT-SoVITS、Piper |
| 內容審核 Moderation | OpenAI Moderation API | LlamaGuard 3、Detoxify |
| 音樂/音效 | Suno/Udio API | MusicGen |
| Embedding（相似案例檢索） | OpenAI/Cohere Embedding | BGE、E5（開源 embedding 模型） |

### Router 判斷邏輯（依序評估）
1. **內容敏感度**：涉及真實寵物臉部/個資 → 優先本地開源，降低資料外流風險
2. **時效性**：急件 → 商用 API；批次任務 → 開源排隊處理
3. **成本上限**：單支影片成本 cap，超過自動降級開源模型
4. **Fallback**：商用 API 失敗/逾時 → 自動切開源模型重試

新增/替換 provider 時：在對應 `providers/` 目錄下實作統一介面（輸入輸出 schema 不變），並在 router 設定檔登記路由規則，不修改呼叫端程式碼。

## 8. 多代理人工作流程

| Agent | 主要責任 |
|---|---|
| Profile Agent | 整理寵物基本資料、健康資訊與領養條件 |
| Media Analyst Agent | 分析照片/影片品質與寵物動作，選出最佳素材 |
| Persona Agent | 建立寵物人格、語氣、聲音角色與個性化表達規則 |
| Marketing Agent | 分析受眾、選擇賣點與影片風格（3 種敘事） |
| Script Agent | 產生 30s/15s 旁白、字幕及 CTA（結構化 JSON） |
| Storyboard Agent | 拆解 5-7 個鏡頭，產生各鏡頭生成提示詞 |
| Generation Agent | 呼叫圖像/影片/語音模型（透過 Provider Adapter） |
| Editing Agent | 剪輯、配樂、字幕、貼圖與平台版型合成（FFmpeg） |
| Fact-check Agent | 逐句比對腳本與 Profile，攔截捏造內容 |
| Quality Agent | 檢查外觀一致性、影音品質與完整性（QA 加權評分） |
| Compliance Agent | 檢查授權、個資、不當內容與 AI 標示 |
| Analytics Agent | 分析觀看與領養轉換漏斗，提出內容優化建議 |

> **MVP 不需要真的部署 12 個獨立微服務。** 初期用一個 Workflow Orchestrator 搭配數個結構化 Prompt 模擬各 Agent 職責即可，待規模增加後再拆成獨立服務。

## 9. MVP 實作邊界（避免一開始就做「完全生成式影片」）

素材組成建議比例：

```
60% 真實影片剪輯
20% 真實照片動態化 (Image-to-Video)
10% 字幕、資訊卡與可愛貼圖
10% AI 生成過場或生活想像場景
```

理由：兼顧寵物真實性、可信度、生成成功率、成本、開發時程與動保單位接受度。

### 最小可行輸入（每隻寵物至少）
3 張清楚照片、1 段 5 秒以上影片、基本資料、3 個個性標籤、1 個需揭露的照護條件、領養聯絡頁面。

### 最小可行輸出
30 秒 9:16 MP4、15 秒精簡版、封面圖、影片標題、社群貼文文案、字幕檔、素材與生成紀錄。

## 10. 任務狀態機

```
DRAFT
  ↓
PROFILE_VALIDATED
  ↓
MEDIA_ANALYZED
  ↓
SCRIPT_GENERATED (3 種風格)
  ↓
SCRIPT_APPROVED
  ↓
ASSETS_GENERATING (鏡頭級 Job，各自獨立)
  ↓
EDITING
  ↓
QUALITY_REVIEW
  ├── FAILED → 只重生失敗鏡頭的 Job（回到 ASSETS_GENERATING，其餘鏡頭不動）
  └── PASSED
        ↓
HUMAN_APPROVED
        ↓
PUBLISHED
        ↓
PERFORMANCE_TRACKED
```

### 設計原則
- 影片生成是非同步工作：每個鏡頭獨立建立 Job，失敗只重生失敗鏡頭
- 保存 Prompt、Seed、模型版本及輸入素材，供追溯
- 設定最大重試次數與成本上限
- 支援暫停、取消、重新生成
- 每支最終影片都能追溯到使用的素材與生成紀錄（對應 Runway 等 API 的非同步任務型態設計）

## 11. 品質檢查機制（QA）

### 自動 QA 指標
- **寵物一致性**：毛色/花紋/耳朵/尾巴/腿數是否異常、項圈與辨識特徵是否保留、是否突然變成其他品種、同影片不同鏡頭是否像同一隻寵物
- **內容真實性**：年齡/性別/健康資料正確性、是否捏造技能、是否隱藏必要照護條件、CTA 連結是否對應正確寵物
- **影音品質**：黑畫面/閃爍/破圖、鏡頭跳動、字幕超出安全區、旁白截斷、總長度偏差、音樂蓋過旁白、開頭吸睛度、結尾 CTA 停留時間

### 加權評分

| 指標 | 權重 |
|---|---|
| Pet Identity Consistency | 30% |
| Factual Correctness | 25% |
| Visual Quality | 15% |
| Audio & Subtitle Quality | 10% |
| Emotional Appeal | 10% |
| Platform Compliance | 10% |

**低於 80 分不進入人工發布畫面；事實正確性或授權/合規檢查任一項失敗，無論總分多少都不可發布。**

## 12. 人工審核介面

正式發布前必須保留人工確認，介面建議三欄式：

- **左側**：原始 Profile 與照片
- **中央**：30 秒影片播放器
- **右側**：分鏡、旁白、QA 警告與修改選項

審核者可：對照原始素材與生成畫面、修改旁白文字、更換聲音角色、關閉某幻想畫面、切換真實素材優先模式、單獨重新生成某鏡頭、調整字幕位置、確認照護條件與聯絡方式、選擇是否標記 AI 生成、核准或退回。

## 13. 字幕與可愛元素視覺設計

### 字幕原則
預設燒錄字幕（靜音也能理解）；每行不超過 12-16 個中文字，最多兩行；關鍵詞可放大（如「兩歲」「已結紮」「等待領養」）；不可遮住寵物臉部；背景複雜時加半透明底板或描邊；CTA 至少停留 2 秒。

### 可愛元素落地技術點
慢動作/凍格特寫高光片段、可愛貼圖疊加（肉球/愛心/星星/搖尾提示線/貓耳狗耳資訊卡）、暖色調 LUT、卡通字幕動畫、BGM 節奏對齊剪輯點（beat-sync cut）。

**不要讓生成模型直接在畫面內產生中文字**——一律由後製引擎（FFmpeg overlay）統一疊加字幕與文字元件，避免錯字與字型變形。

Persona Agent 輸出規格：`pet_name`、`personality_tags`、`voice_style`、`catchphrase`、`cute_moments`（供剪輯層標記高光片段的 `shot_type: cute_highlight`）。

## 14. 資料、安全與倫理設計

- 上傳者必須擁有素材使用與 AI 生成授權
- 人臉、地址、電話、車牌、醫療文件需去識別化
- 不得模仿未授權真人聲音
- 不得虛構寵物健康、行為或領養資格
- AI 創意畫面須清楚標示
- 保存生成紀錄與發布版本，供事後追溯與合規查核
- 提供撤下影片及刪除素材功能
- 聯絡資料使用短網址或平台頁面，避免直接露出個人電話
- 不使用悲情操弄、死亡倒數或威脅式文案提高點擊
- 影片最終發布必須由收容單位或送養人確認

## 15. 成效分析與閉環優化

### 全漏斗指標
```
曝光 → 3秒停留 → 完播 → 點擊寵物頁面 → 諮詢 → 申請 → 會面 → 領養成功
```

| 面向 | 指標 |
|---|---|
| 生成效率 | 平均完成時間、人工編輯時間、失敗重生率 |
| 影片品質 | 審核通過率、寵物一致性分數、字幕錯誤率 |
| 觀看效果 | 3秒留存率、完播率、重播率 |
| 互動效果 | 按讚、分享、留言、收藏率 |
| 領養效果 | 寵物頁點擊率、諮詢率、申請率、領養率 |
| **公平性** | 不同年齡/毛色/品種/健康狀態寵物的曝光差異 |

### A/B 測試維度
萌系 vs. 溫暖故事 vs. 反差幽默、第一人稱 vs. 第三人稱、真實影片優先 vs. 輕度動態化、男聲 vs. 女聲、「認識我」CTA vs. 「我要領養」CTA、30 秒 vs. 15 秒。

Analytics Agent 的優化目標是「**有效諮詢與適配領養**」，不是單純曝光/觀看數，避免演算法系統性偏好幼犬幼貓或特定外觀（公平性監控見上表）。

## 16. 開發階段規劃

### 第一階段：PoC（概念驗證）
單一寵物 Profile、上傳 3-10 張照片＋1-3 段影片、產生一版腳本、第一人稱中文旁白、以真實影片＋照片 Ken Burns 動態效果為主、自動字幕/音樂/CTA、匯出 1080×1920 MP4、人工核准下載。

**驗證目標**：30 秒內能否完整說清楚寵物亮點、旁白是否自然、製作時間是否明顯低於人工剪輯、收容單位是否願意採用、寵物資訊是否保持正確。

### 第二階段：MVP
多寵物管理、3 種腳本風格、Image-to-Video、多種聲音角色、分鏡編輯、單鏡頭重新生成、影片 QA、模板管理、多尺寸輸出、生成成本追蹤、審核與版本歷程。

### 第三階段：正式產品
多組織與權限管理、社群發布整合、成效儀表板、A/B 測試、品牌模板、多語言（含台語）、領養諮詢聊天機器人串接、內容自動推薦、成功案例知識庫、模型路由與成本最佳化、素材授權與稽核紀錄、對外 API。

> 團隊規模僅供擴編參考，非必要編制：PM/領養流程顧問 1 人、AI/後端工程師 1-2 人、前端工程師 1 人、影音工程師 1 人、UI/UX 0.5-1 人、QA 0.5-1 人、動保業務顧問兼任。

## 17. 首版系統流程（端到端）

```

12. 輸出影片及發布
13. 回收觀看與領養轉換數據

```
